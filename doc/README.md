# block-agent 使用者文档

这套文档想帮你回答三个问题：block-agent **是什么**、它为什么**这样设计**、以及你怎么**用它、扩展它**。读完你会知道怎么把内置能力跑起来、怎么管理这些能力、以及怎么写一块属于你自己的能力装进去。

## 一句话：它是什么

block-agent 是一个通用的 agent 运行时。它把"进入模型的那段上下文"当成第一性的东西来经营——不是一坨拼出来的字符串，而是**一份有结构的上下文**，由许多可以独立装卸的能力模块共同组成。每一片功能都是一个 **BlockApp**：对话历史是一个 BlockApp，工具是一个，记忆是一个，agent 的身份也是一个。

关键在于**没有"内核特权模块"**。内置的能力和你写的第三方能力走的是完全相同的形态、相同的入口、相同的约束。这让运行时可以一块一块地生长，而不是越长越像一个谁也不敢动的黑盒。

## 三个贯穿全局的想法（读任何文档前先记住）

这三条不是技术细节，是理解全部设计的钥匙：

- **有结构的上下文**。模块不直接往提示词里塞字符串。每个模块只负责把自己的状态**呈现**成上下文里的一片，运行时每一轮把这些片拼成完整上下文。这个呈现过程是确定的——同样的状态总是呈现出同样的内容，这正是上下文缓存能稳定命中的前提。

- **受约束的操作是唯一的写入口**。agent 想做任何事，都必须通过一次明确的**操作**（一次结构化的命令调用）。光说一段话不算动作——会被拒绝、并作为错误反馈回去。使用者、agent、模块三方都走同一组操作、同一道统一的鉴权关口，没有谁有后门。

- **模块按声明的契约协作，而不是互相点名**。一个模块说"我提供某种能力/数据"，另一个说"我需要这种能力/数据"，双方都不需要知道对方是谁。换掉一个模块，依赖它的那些模块不用改一个字。

## 一分钟跑起来

```bash
# 在仓库根目录。用真实模型需要一个 key（从环境变量读，绝不写进配置文件）：
$env:ANTHROPIC_API_KEY = "<你的 key>"          # PowerShell
export ANTHROPIC_API_KEY=<你的 key>            # bash
npm start

# 没有 key 也想看它跑：用离线的 mock 模型（脚本化地回一句话）
npm start -- --dry-run
```

启动后是一个交互式终端：直接打字 = 给 agent 发消息；以 `/` 开头 = 斜杠命令（`/help` 看全部）。

## 一句话总结这套设计

灵活，是因为模块绑的是契约不是身份；安全，是因为所有写入都收敛到唯一一道关口；清晰，是因为上下文是确定地呈现出来的。这三件事合起来，就是 block-agent 想给 agent 的那份"结构清晰的上下文"。
